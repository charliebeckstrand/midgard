# React 19.2 Feature Adoption Audit — `ui`

**Date:** 2026-07-03 · **Scope:** the three features React 19.2 shipped — `<Activity>`, `useEffectEvent`, and `cacheSignal` ([release notes](https://react.dev/blog/2025/10/01/react-19-2)) — weighed against the whole `ui` surface (`components/`, `primitives/`, `modules/`, `hooks/`, `providers/`, `layouts/`). React **19.2.7** is installed and all three symbols resolve from the client build; the package peers `react` / `react-dom` at `^19.2.0` ([`package.json`](../../package.json)). **Method:** four parallel single-lens sweeps ([CLAUDE.md](../../../../CLAUDE.md) §3.3) — `cacheSignal` applicability, `<Activity>` structure, and `useEffectEvent` across `{hooks, primitives, providers}` and `{components, modules}` — each returning `file:line`-grounded findings; the Biome interaction was settled empirically by running throwaway probe files through the real `biome check`; every load-bearing claim was re-read against source.

**This is an audit — no code, types, or components changed.** It is a point-in-time, single-lens record per [CONVENTIONS.md](../../../../CONVENTIONS.md) §12.3, and a living one: adoption rows resolve in place as the recommended migrations land.

---

## Executive summary

| Feature | Verdict | Where it lands |
|---|---|---|
| `cacheSignal` | **Not applicable.** | React-Server-only API; the package is client-only with no `cache()` surface. Nothing to integrate. |
| `<Activity>` | **Already adopted, correctly — extend to 3 targets.** | Live in the `current` primitive (Tabs/Nav/Segment). Best new target: **Stepper**; then **Collapse** and **Accordion** as opt-ins. |
| `useEffectEvent` | **Adopt, narrowly (~11 internal sites).** | The dismiss/floating/keybinding hooks and a handful of controlled-input state hooks already hand-roll the exact idiom it supersedes. Not a sweeping refactor. |

Two of the three features are worth acting on, neither as a broad rewrite. `<Activity>` is the more strategic — the library already owns a correct reference implementation (`primitives/current`), so extension is low-risk reuse rather than new ground. `useEffectEvent` is a cleanup lever: it deletes a hand-rolled `const xRef = useRef(x); xRef.current = x` (and sometimes a whole sync Effect) at each genuine site, and Biome already enforces its one mechanical rule. `cacheSignal` has no surface here and is excluded with evidence, not assumption.

The single most important discrimination in this audit: **most of the library's latest-ref usage is correctly _not_ `useEffectEvent` material.** Virtualized data-cache refs, previous-value trackers, `AbortController`/timer/disposer instances, and — the largest trap — callback refs whose consumers are JSX-attached DOM handlers or other hooks are all disqualified, because an Effect Event may only be called from inside an Effect. The recommended set is the residue that survives that filter.

## 1. `cacheSignal` — not applicable

`cacheSignal()` returns an `AbortSignal` tied to the lifetime of a React Server Components `cache()` call, so server-side async work can be cancelled when React discards the cache. It exists only in the `react-server` build condition and is meaningless in a client component.

This package has **zero** server surface, established by evidence rather than inference:

- No `react-server` export condition anywhere in [`package.json`](../../package.json) `exports` (every entry exposes only `types` / `default`), so the `react-server` build — the only place `cacheSignal` and `cache()` live — is never selected.
- No `'use server'`, no `server-only` import, no `react-dom/server` render path, and no async React components across `src` (grep: no matches); `tsup.config.ts` builds a single client ESM bundle.
- `'use client'` is pervasive (~250 files); `cache(` / `cacheSignal` appear nowhere in source (the lone `cache` hit is an unrelated hand-rolled LRU comment in [`code-block.tsx`](../../src/components/code/code-block.tsx)).

A precise nuance, not a loophole: the library deliberately keeps a **static, server-renderable leaf** tier ([REFERENCE.md](../../REFERENCE.md) §2) — components with no `'use client'`, no hooks, no context (pinned by `static-component-boundary.test.ts`, e.g. `avatar`, `badge`, `card`, `table`, `markdown`). These are *shared* components: they render **inside** a consumer's RSC tree, but they are synchronous and hook-free and never call `cache()` — so there is still nothing for `cacheSignal` to attach to.

Nor is `cacheSignal` a substitute for the client cancellation the package already does correctly. Every async site owns its lifetime through an `AbortController` or a caller-supplied `AbortSignal` — a different lifetime owner (effect cleanup / user `stop()`), not an RSC cache boundary:

| Site | Async work | Cancellation today | `cacheSignal`? |
|---|---|---|---|
| [`use-address-input-suggestions.ts`](../../src/components/address-input/use-address-input-suggestions.ts) | debounced geocoding via injected provider | `AbortController` per run, aborted on re-query / unmount | No — client hook |
| [`use-chat-send.ts`](../../src/modules/chat/use-chat-send.ts) | streamed agent reply via injected transport | `AbortController`; `stop()` aborts | No — client hook |
| [`map-routing.ts`](../../src/modules/map/map-routing.ts) | OSRM / Valhalla `fetch` | caller-supplied `AbortSignal` | No — framework-free browser `fetch` |
| [`use-pdf-viewer-document.ts`](../../src/components/pdf-viewer/use-pdf-viewer-document.ts) | dynamic `pdfjs` import + render | bespoke `cancelled` flag + `renderTask.cancel()` | No — effect-owned teardown |
| [`code-block.tsx`](../../src/components/code/code-block.tsx) · [`markdown.tsx`](../../src/components/markdown/markdown.tsx) | lazy `shiki` import · synchronous `marked` lex | module-cached / N/A (synchronous) | No |

**Verdict:** not applicable, and unambiguously so. The single root reason is that `cacheSignal` is bound to an RSC `cache()` scope this code never enters. It would only become relevant if the package began shipping RSC **data** components (async, `cache()`-fetching) under a new `react-server` condition — a shape it explicitly avoids.

## 2. `<Activity>` — already adopted; extend to three targets

`<Activity mode="visible" | "hidden">` keeps children **mounted with state preserved** while hidden, but hides them via `display: none`, tears down their Effects, and renders them at lower priority; switching back restores state and re-runs Effects. Two caveats bound its use: `display: none` **cannot animate**, and it does **not** remove the DOM (so it is not a substitute for teardown that a focus trap, portal, or heavy DOM actually needs).

### 2.1 Existing adoption is correct

`<Activity>` lives in exactly one place — [`current-content.tsx`](../../src/primitives/current/current-content.tsx) (`<Activity mode={current ? 'visible' : 'hidden'}>`) — and is reached only through a deliberate funnel: the default `mount="active"` still fully unmounts inactive panels, and Activity wraps a panel **only** when it is a *held* inactive panel (`mount` is `always`/`lazy`) **and** `fade` is off. The animating path never touches Activity; held panels cross-fade through a `motion.div` opacity/position transition instead — precisely because `display: none` can't animate. The tradeoff is documented in the primitive's TSDoc and pinned by tests (`current.test.tsx`, `tabs.test.tsx`): DOM-present-but-hidden, uncontrolled state preserved across switches, Effects torn down then re-run on show, and lazy defer-then-hold.

The abstraction is generic and **already reused**: [`current`](../../src/primitives/current) is parameterized by `slotPrefix` and backs **Tabs**, **Nav** ([`nav-content.tsx`](../../src/components/nav/nav-content.tsx)), and **Segment** (`<Tabs variant="segment">`). No fix is owed here.

### 2.2 New candidates

| Component | Current gating | Classification | Rationale |
|---|---|---|---|
| **Stepper** ([`stepper-panel.tsx`](../../src/components/stepper/stepper-panel.tsx)) | `if (value !== currentValue) return null` | **Adopt (opt-in)** | Canonical wizard case. Stepping backward unmounts the outgoing panel, resetting uncontrolled inputs, scroll, and expensive subtrees. Bare `<section>` — no animation, portal, or trap to conflict. |
| **Collapse**, non-animated path ([`collapse-panel.tsx`](../../src/components/collapse/collapse-panel.tsx)) | `open ? <section> : null` (`animate={false}`) | **Offer as option** | The `animate={false}` seam already exists as a clean drop-in; a keep-mounted opt-in preserves in-panel drafts/scroll with no transition to fight. |
| **Accordion** ([`accordion-panel.tsx`](../../src/components/accordion/accordion-panel.tsx)) | `{open && <motion.div height/opacity>}` | **Conditional opt-in** | Useful for stateful bodies, but requires first introducing an unanimated keep-mounted mode; per-item Activity is valid even under `type="multiple"` since each item is an independent disclosure. |

Everything else that mounts-on-open is **correctly unmounting** and must stay that way: `dialog`, `drawer`, `sheet` (via [`overlay.tsx`](../../src/primitives/overlay/overlay.tsx)), `popover` and every floating dropdown — `listbox`, `combobox`, `select`, `date-picker`, `color-picker` — (via [`floating-surface.tsx`](../../src/primitives/floating-surface/floating-surface.tsx)), and `command-palette`. Each pairs a portal with a focus trap and an exit animation; they must leave the DOM for focus return and to avoid stranded portal nodes. `segment` already routes through Tabs' `mount` prop; `sidebar` mini-rail hides labels in place via CSS (nothing unmounts); `pdf-viewer` already hides its rail with an animatable margin-slide + `inert` (Activity would *regress* it) and must not keep rasterized page images mounted; `pivot-table` and `filters` gate only stateless, prop-derived nodes.

### 2.3 Reuse over per-component work

The highest-leverage move is to route **Stepper** through the existing `current` primitive, exactly as Nav and Segment already are: make `StepperPanels`/`StepperPanel` thin adapters over `CurrentContents`/`CurrentContent` (keyed on `String(value)`). That hands Stepper `mount="always"/"lazy"` + Activity **and** the fade height animation **and** the shared a11y-disclosure wiring for free, instead of a bespoke implementation.

Do **not** route Accordion/Collapse through `current` — wrong shape. `current` models "one active view of N"; Accordion `type="multiple"` allows many independent open disclosures, and both animate per-panel height `0→auto`, which `current`'s equal-height opacity cross-fade doesn't represent. For these, a per-component `keepMounted`-style opt-in that swaps `{open && …}` for `<Activity>` **only on the non-animated path** mirrors how Tabs already confines Activity behind `fade={false}`.

### 2.4 Anti-patterns (each grounded in this codebase)

- Never wrap an `AnimatePresence` / height- or opacity-animated panel in Activity — the animated paths of Accordion and Collapse, and every overlay/dropdown, animate on hide; `display: none` kills the transition. The `current` primitive already models the correct escape hatch (fade → opacity; no-fade → Activity).
- Never replace a portal + focus-trap surface with Activity — dialog/drawer/sheet, popover/dropdowns, and command-palette must leave the DOM for focus restoration and to avoid empty portal nodes.
- Never keep memory-heavy DOM mounted-but-hidden — the pdf-viewer's rasterized page images.
- Don't reach for Activity where an in-place CSS hide is already better — the pdf rail's margin-slide + `inert` and the sidebar mini-rail's `sr-only` labels both animate and preserve state, which Activity cannot.

## 3. `useEffectEvent` — adopt narrowly

`const onX = useEffectEvent(fn)` extracts non-reactive logic from an Effect: `onX` always sees the latest props/state but is omitted from — and never re-triggers — the Effect's dependency array. It is the native replacement for the userland "latest ref" idiom (`const cbRef = useRef(cb); cbRef.current = cb;` read by a subscription that must not re-subscribe). Its binding constraint: an Effect Event may **only be called from inside an Effect** (or a listener/observer/timer created in one) — never during render, never from a JSX-attached DOM handler, never passed to another component or hook.

The library hand-rolls this exact idiom, concentrated in the dismiss/floating/keybinding subsystems (which all subscribe through a shared `document-listener`) and a few controlled-input state hooks. Applying the constraint above leaves **~11 genuine sites**.

### 3.1 Candidates worth converting

| Site | Current pattern → what it becomes | Risk |
|---|---|---|
| [`use-escape-layer.ts`](../../src/hooks/use-escape-layer.ts) | `onDismissRef` feeds the shared keydown listener → `onDismiss` Effect Event | **low** — single consumer; foundational for the whole dismiss stack |
| [`use-floating-disclosure.ts`](../../src/hooks/use-floating-disclosure.ts) | `onOpenChangeRef` feeds the outside-press listener → Effect Event | **low** — single consumer, internal-only |
| [`use-dismissable.ts`](../../src/hooks/use-dismissable.ts) | `onDismissRef` → Effect Event for the pointerdown listener | **low–med** — two consumers; pass the *raw* prop to `useEscapeLayer`, not the Event |
| [`use-floating-ui.ts`](../../src/hooks/use-floating-ui.ts) (`useFloatingUI`) | `onOpenChangeRef` → Effect Event for the pointerdown listener | **low–med** — same two-consumer split as above |
| [`use-keybindings.ts`](../../src/hooks/use-keybindings.ts) | `bindingsRef` (map) → one dispatch-by-key Effect Event feeding tinykeys | **low–med** — highest-traffic subscription; needs a `dispatch(key, e)` indirection |
| [`use-grid-column-auto-size.ts`](../../src/modules/grid/use-grid-column-auto-size.ts) | `runRef` synced in a layout Effect, read only by the `ResizeObserver` → `onAutoSize` Event | **low** — the code comment already states the exact rationale; `run` stays a `useCallback` for its direct callers |
| [`use-grid-editing.ts`](../../src/modules/grid/use-grid-editing.ts) | `onValueChangeRef` read only by the flush Effect → Effect Event | **low** — pure Effect; the clean half of the grid |
| [`use-password-confirm-state.ts`](../../src/components/password-confirm/use-password-confirm-state.ts) | `onMatchRef` / `onMismatchRef` → two Effect Events in one transition Effect | **low** — pure Effect |
| [`use-password-strength.ts`](../../src/components/password-strength/use-password-strength.ts) | `onChangeRef` read only by the report Effect → Effect Event | **low** — pure Effect |
| [`credit-card-input-cvv.tsx`](../../src/components/credit-card-input/credit-card-input-cvv.tsx) | a `{value, setValue, onValidityChange}` `latestRef` bundle read only in the brand-change Effect → one Effect Event | **low** — deletes the whole bundle ref |
| [`use-address-input-suggestions.ts`](../../src/components/address-input/use-address-input-suggestions.ts) | `providerRef` + its sync Effect → one `callProvider` Effect Event in the debounce timer | **low–med** — deletes a ref *and* a sync Effect; timer created in the Effect (sanctioned) |

One further public-API site is worth a **deferred, deliberate** conversion: [`use-resize-observer.ts`](../../src/hooks/use-resize-observer.ts) takes `callback` as an Effect dependency, so its TSDoc currently forces callers to `useCallback` and hand-roll their own latest-ref or the observer re-subscribes each render. Internalizing `const onResize = useEffectEvent(callback)` removes that caller-side footgun — and, as a bonus, is the correct home for the [`use-signature-pad-canvas-sizing.ts`](../../src/components/signature-pad/use-signature-pad-canvas-sizing.ts) styling-bundle ref, whose real fix lives here, not at the call site. It ranks last because it is a public contract change (a changed callback would no longer re-subscribe/re-fire), which pulls in the [CLAUDE.md](../../../../CLAUDE.md) §3.5 TSDoc + `docs/*` obligation and an audit of out-of-scope callers that rely on re-fire-on-change.

### 3.2 Explicitly excluded — looks like a candidate, isn't

The bar rejects far more than it admits; recording the categories so a future pass need not re-derive them:

- **Callback refs read by JSX-attached DOM handlers** (the biggest trap). The grid's most-cited refs — `onRowActivateRef`, `toggleActiveRowRef`, `activeRef` in [`use-grid-navigation.ts`](../../src/modules/grid/use-grid-navigation.ts) — are read inside `onKeyDown`/`onFocus` `useCallback`s that are spread onto the `<table>` via `navTableProps`. They keep a JSX handler's deps stable; they are **not** Effect-latest refs, and an Effect Event may not be attached to a DOM handler. Same shape: `onChangeRef`/`formatRef` in [`use-color-state.ts`](../../src/components/color/use-color-state.ts), `onValueChangeRef` in [`use-accordion-selection.ts`](../../src/components/accordion/use-accordion-selection.ts), the reducer callback refs in [`use-form-reducer.ts`](../../src/components/form/use-form-reducer.ts), `onSizesChangeRef` in [`use-resizable-panel.ts`](../../src/components/resizable/use-resizable-panel.ts).
- **Callback refs read from a returned setter/handler, not an Effect.** `onValueChangeRef` in [`use-controllable.ts`](../../src/hooks/use-controllable.ts), `gateRef` in [`use-floating-disclosure.ts`](../../src/hooks/use-floating-disclosure.ts), and `onOpenChangeRef` inside `useFloatingPanel` ([`use-floating-ui.ts`](../../src/hooks/use-floating-ui.ts)) stabilize a callback for an event-time path. Converting these would violate React's contract — **and Biome would not catch it** (§4). Leave them.
- **Virtualized data-cache refs, read during render.** `rowsRef`, `colCountRef`, `rowIndexMapRef`, `rowKeysRef`, `dataColumnsRef`, `columnsByIdRef` across [`grid-data.tsx`](../../src/modules/grid/grid-data.tsx) and siblings hold data, not callbacks, and are read at render time — outside the mechanism entirely.
- **Instance / timer / disposer refs, and prev-value / flag trackers.** `AbortController` refs (chat, address-input), timeout ids, animation-control handles, cleanup-fn disposers, mount-skip flags, and previous-value trackers (`prevOpenRef`, `wasDisabledRef`, dedupe caches) are all out of category — some are read in Effects, but there is no reactive callback to make non-reactive.
- **Already-idiomatic subscriptions.** [`use-media-query.ts`](../../src/hooks/use-media-query.ts) and [`use-offcanvas.ts`](../../src/hooks/use-offcanvas.ts) use `useSyncExternalStore` / `matchMedia` with correctly-reactive inputs and stable setters — no callback ref, nothing to convert.

Three sites are genuinely **borderline and deferred**: [`use-hold-button-gesture.ts`](../../src/components/hold-button/use-hold-button-gesture.ts) (`cancelRef`) and [`use-copy-button-state.ts`](../../src/components/copy-button/use-copy-button-state.ts) (`onCopiedChangeRef`) each have one sanctioned Effect consumer *and* an event-handler co-consumer, so the ref can't be fully removed without routing an event-time call through the Event — outside the strict rule.

## 4. Tooling & constraints — verified empirically

Biome 2.4.16 (the repo's only hooks linter — there is no `eslint-plugin-react-hooks`) was probed with throwaway files run through the real `biome check --config-path=../..`:

- Biome **recognizes** `useEffectEvent` imported from `react` and treats its return as non-reactive. Omitting the Event from an Effect's deps lints **clean** — **no false positive from adoption**.
- Adding the Event to a dep array **errors** (`useExhaustiveDependencies`, "specifies more dependencies than necessary") with an autofix to remove it — so Biome actively **enforces** the omit-from-deps rule.
- **Gap:** Biome does **not** enforce "only call an Effect Event from inside an Effect" — calling one from a `useCallback` lints clean. This is exactly the failure mode of the §3.2 setter-refs, and with no `eslint-plugin-react-hooks` in the repo, **reviewer discipline is the only guard**. Every conversion must be checked by hand against that rule.

This corrects a naïve static read: grepping Biome's JS wrapper package for the string finds nothing, which would suggest no support — but the compiled engine plainly handles it, as the probes show. Trust the behavior, not the grep.

Adoption also respects the repo's cleanliness posture: the codebase carries only **two** `useExhaustiveDependencies` suppressions today (both intentional, in `modules/`), and none of the recommended conversions add one.

## 5. What was verified

Recorded so a future pass need not re-derive them:

- **React 19.2.7 installed**; `Activity`, `useEffectEvent`, `cache`, and `cacheSignal` all export from the client `react` build (`node -e` on the resolved entry).
- **`cacheSignal` has no surface**: no `react-server` export condition, no `'use server'`, no server render path, ~250 `'use client'` files; all async work is client `AbortController` or synchronous.
- **`<Activity>` is confined correctly**: read of [`current-content.tsx`](../../src/primitives/current/current-content.tsx) confirms Activity wraps only held, non-fade panels; the fade path uses opacity; three consumers (Tabs/Nav/Segment) share the primitive.
- **The grid-navigation refs are DOM-handler-bound**: `onKeyDown`/`onFocus` are `useCallback`s spread via `navTableProps` onto the `<table>` — confirming their exclusion — while [`use-grid-column-auto-size.ts`](../../src/modules/grid/use-grid-column-auto-size.ts)'s `runRef` feeds a `ResizeObserver` created inside an Effect, confirming its inclusion.
- **Biome behavior** confirmed by execution, not memory (§4).

## 6. Recommendation

Two features, sequenced by leverage and risk:

1. **`<Activity>` → Stepper**, implemented by reusing the `current` primitive (§2.3). Highest strategic value, lowest marginal cost — it is reuse of a tested reference, not new machinery. Then Collapse's non-animated opt-in, then Accordion's conditional one.
2. **`useEffectEvent`**, in the §3.1 order — start with the single-consumer dismiss hooks ([`use-escape-layer.ts`](../../src/hooks/use-escape-layer.ts) first; it sets the pattern for the stack), then the two grid Effect sites and the controlled-input hooks. Hold [`use-resize-observer.ts`](../../src/hooks/use-resize-observer.ts) until the internal-only conversions land, since it changes a public contract.
3. **`cacheSignal`** — no action; revisit only if the package ever ships RSC data components.

Each of these is a **multi-file behavioral change** and therefore follow-up work to be surfaced for assent before implementation ([CLAUDE.md](../../../../CLAUDE.md) §3.1), touching TSDoc and the matching `docs/*` surface where a public signature moves (§3.5, [CONVENTIONS.md](../../../../CONVENTIONS.md) §12). This audit is the map, not the migration.
