# ARIA Audit — `packages/ui`

**Date:** 2026-06-08 · **Scope:** the interactive, overlay, composite-widget, and data-grid surface (103 component directories; pure layout primitives — `box`, `flex`, `grid`, `stack`, `container`, `divider`, `spacer`, `aspect-ratio`, `frame`, `card`, `group` — carry no ARIA and are excluded). **Method:** manual source sweep against the WAI-ARIA Authoring Practices Guide (APG), targeting defects a static axe-core render *cannot* catch — focus management, keyboard operability, live-region announcement of dynamic events, dynamic state-attribute wiring, accessible naming, and role/structure correctness. **No component source was modified.**

ARIA here is hand-rolled — no Radix/Ark/Base UI; only `@floating-ui/react` for positioning — so every role, key handler, and live region is the library's own responsibility and is audited as such.

---

## Executive summary

The accessibility foundations are genuinely strong, and several components are textbook. The `Control`/`Field` system composes `aria-describedby` from only-mounted slot ids (via `useA11yScope`), so references never dangle; `aria-invalid`/`aria-expanded`/`aria-selected`/`aria-checked` are bound to live state rather than hardcoded; modal focus management (trap, return, background `inert`, Escape) is real, driven by `FloatingFocusManager modal` under `Dialog`/`Drawer`/`Sheet`. `CommandPalette` is a reference-correct virtual-`aria-activedescendant` combobox-in-dialog; `Combobox` was brought onto the same virtual-roving model and no longer moves DOM focus onto options; `Listbox`/`Select`/`Combobox`/`Menu`/`Popover` correctly pass `role: null` to suppress floating-ui's `useRole` so no positioning wrapper double-stamps a duplicate widget role. `TagInput`, `FileUpload`, `Tooltip` (WCAG 1.4.13), `Alert`/`Banner` severity→live-region mapping, `PasswordStrength`, `DataTable` (`aria-sort`, virtualized row indices, labeled selection), `Table`/`PivotTable` semantics, `Nav`/`NavList`, and the `pdf-viewer` are all clean.

The remaining findings cluster into a handful of **systemic patterns** rather than scattered one-offs. Resolving the cross-cutting themes below clears the majority of individual rows.

### Headline issues

1. **Keyboard operability gaps that lock users out entirely.** `SignaturePad`'s canvas is `tabIndex={-1}` and pointer-only — no keyboard path to sign (WCAG 2.1.1) — *still open*. ~~A `loading` `Button` rendered as an anchor keeps `href` live and `onClick` wired, so it still navigates despite `aria-disabled`.~~ **✅** (gated: out of tab order + activation cancelled). ~~The idiomatic `TabContents`/`TabContent` API emits panels with no `role="tabpanel"`, `aria-labelledby`, or `tabIndex`.~~ **✅** (auto-wired tabpanel + reciprocal links + content-aware focusability). (Critical)
2. **Dynamic changes that never reach assistive tech.** ~~Keyboard drag-and-drop in `Kanban` and `List` emits no announcements — lift/move/drop are silent.~~ **✅** (both now announce lift/move/drop with card/column names + position). ~~`Calendar`/`DatePicker` month navigation~~ **✅** (the Calendar announces `monthLabel` politely on view change). ~~Still open: `NumberInput` stepper buttons~~ **✅** (stepped values announced). ~~`Filters` active-count~~ **✅** (announced politely on change). ~~Polite `Toast` severities~~ **✅** (mirrored through the announcer on mount). (Critical/High, WCAG 4.1.3)
3. **Roving-tabindex theater. ✅ RESOLVED.** `Toolbar`, `Stepper`, `Sidebar`, `Breadcrumb`, and `Pagination` wired `useA11yRoving` and advertised a single-tab-stop model, but the hook never wrote `tabIndex` — only `TabList` (MutationObserver) and `Tree` initialized it — so Tab visited every control while arrows *also* moved focus, and the nav surfaces imposed roving on ordinary site-nav links against `Nav`'s deliberate model. `useA11yRoving` now offers opt-in `manageTabIndex` (+ `activeSelector`) that owns the resting single stop and carries it with focus; `Toolbar` and `Stepper` enable it (genuine composite widgets), `Sidebar` is reimplemented as true roving seated on the current page, and `Breadcrumb`/`Pagination` drop roving entirely to match `Nav`. (High)
4. **Composite roles shipped without an accessible name. ✅ RESOLVED (naming half).** `tablist`, `toolbar`, `tree`, `radiogroup`, the `listbox`/`combobox` popups, `kanban` columns, `pivot-table`, and the `filters` bar emitted a role with no defaulted or required name. Names are now enforced (`Toolbar`/`Tree`/`TabList`/`RadioGroup`/`Filters` require `aria-label`/`aria-labelledby` via the shared `AccessibleName` type) or threaded (listbox/combobox popups inherit the trigger name; kanban columns name themselves from their title; `PivotTable` takes an optional caption). The separate placeholder-as-label cluster (credit-card expiry/CVV, `query-builder` date value) remains — see pattern D. (High)
5. **State conveyed by color/CSS only, invisible to AT.** ~~`Stepper` completed-vs-upcoming~~ **✅** (sr-only per-step status). ~~`EditableGrid` range selection~~ **✅** (verified fixed — `aria-selected` driven from the range slice). `Resizable`'s `aria-orientation` reports the layout axis instead of the separator's own (perpendicular) axis. (High)

## Cross-cutting patterns

**A. Roving-tabindex claimed but never delivered (High). ✅ RESOLVED.** `useA11yRoving` moved DOM focus on Arrow/Home/End but never assigned `tabIndex`, so single-tab-stop only held where a component initialized it separately (`TabList`/`Tree`); `Toolbar`, `Stepper`, `Sidebar`, `Breadcrumb`, and `Pagination` shipped every item at native `tabIndex=0`. The hook now takes an opt-in `manageTabIndex` (focus mode): a `MutationObserver` seats exactly one matched item at `tabIndex=0` (the `activeSelector` match, else the first) with the rest at `-1`, writing only on divergence so it never feeds back on its own edits, and `moveTo` carries the `0` with focus on each arrow. `TabList`/`Tree` are untouched (default off). The composite widgets opt in: `toolbar/toolbar.tsx` (`manageTabIndex`), `stepper/stepper.tsx` (`manageTabIndex` + `activeSelector: '[aria-current="step"]'`), `sidebar/sidebar.tsx` (`+ '[aria-current="page"]'`). The link surfaces drop roving (pattern B).

**B. Nav-surface keyboard-model inconsistency (High). ✅ MOSTLY RESOLVED.** `Nav`/`NavList` (`nav/nav-list.tsx:18-23`) deliberately keep site-nav links individually Tab-focusable. `Breadcrumb` and `Pagination` previously roped their links into arrow roving; both now drop `useA11yRoving` and render ordinary Tab-through links, matching `Nav`. `Sidebar` intentionally *keeps* a roving model (app-style vertical nav) but reimplemented correctly via pattern A — a real single Tab stop seated on the current page, not a half-roving hybrid. ~~*Still open:* `BottomNav`→`Nav` ships an unnamed `<nav>` landmark~~ **✅ RESOLVED** — `BottomNav` defaults `aria-label="Bottom"` (overridable), matching `Navbar`'s pattern.

**C. Roles without an enforced accessible name (High/Medium). ✅ RESOLVED.** A role was emitted but no name was defaulted or required. The fix split two ways. **Require** (a new shared `types/AccessibleName` — `{ 'aria-label': string } | { 'aria-labelledby': string }`, the union `ToggleIconButton`/`ProgressBar`/`ProgressGauge` already used inline, now hoisted): `Toolbar`, `Tree`, `TabList` (so `SegmentControl` too), `RadioGroup`, and `Filters` (newly a `role="group"`) intersect it, so an unnamed instance is a compile error. **Thread** (no API change): the `listbox`/`combobox` popups take their name from the trigger/input (`PopoverPanel` and the combobox inner list accept `aria-label`/`aria-labelledby`, fed from the resolved control name), and `kanban` columns name their `<section>` from the rendered `KanbanColumnTitle` via `aria-labelledby` (explicit `aria-label` wins). **Optional**: `PivotTable` accepts an `aria-label` caption (a native `<table>` is valid unnamed). The `Field`/`Label`-wrapped case is also covered: `useA11yControl` gained a registered `label` slot (relation `labelledby`), so `Control`/`Field` expose the Label's id as `labelledBy`, and the listbox/combobox popups fall back to it when no explicit name is passed — a portalled popup now names itself from the field's `<Label>` with no dangling reference. All internal callers, demos, and tests updated.

**D. Placeholder-as-label (High).** Placeholder is not a programmatic name and vanishes on input. Affected: ~~`credit-card-input/credit-card-input-cvv.tsx:70`, `credit-card-input-expiry.tsx:37`~~ **✅** (default `aria-label`s that yield to a Field label), `query-builder/query-builder-rule-value.tsx:61` (the `DatePicker` date branch — siblings pass `aria-label`, this one does not). Bare `Input`/`Textarea`/`Combobox`/`Listbox` outside a `Field` are also unnameable except via an optional `aria-label` that nothing enforces. **Fix:** explicit `aria-label`s (or mandate `Field`/`Label` wrapping with a dev-time assertion).

**E. State conveyed by color/CSS only (High, WCAG 1.4.1).** Affected: ~~`stepper/stepper-step.tsx:93` + `stepper-indicator.tsx:22`~~ **✅** (sr-only per-step status), ~~`editable-grid/editable-grid-cell.tsx:48`~~ **✅** (verified fixed — `useEditableGridCellAriaSelected` drives `aria-selected` from the same slice). **Fix:** add a visually-hidden status suffix / `aria-label`, and `aria-selected` driven by the same slice that sets `data-in-range`.

**F. Dynamic events with no live-region announcement (High/Medium).** ~~`kanban/use-kanban-keyboard.ts` and `list/use-list-keyboard.ts` (hand-rolled keyboard reorder, zero announcements)~~ **✅** now announce lift/move/drop. Still affected: `calendar/use-calendar-month.ts` + `date-picker/use-date-picker-keyboard.ts` (month change silent), `number-input/number-input.tsx:130` (stepper-button value silent), ~~`filters/filters.tsx:82`~~ **✅** (active count announced politely on change), `toast/toast-alert.tsx:76` (polite severities silent), `hold-button/use-hold-button-gesture.ts:54` (no completion announcement). The `announce()` / `useA11yAnnouncements` infrastructure already exists — feed it the relevant string.

**G. Dangling `aria-controls` on unmount-while-closed disclosures (Medium). ✅ RESOLVED.** ~~`accordion/accordion-panel.tsx:20` and `collapse/collapse-panel.tsx:18`~~ — both triggers now gate `aria-controls` to the open state (the Stepper pattern, `stepper-step.tsx:96`), so the reference exists only while its target panel is mounted.

**H. Custom value-widgets miss Page keys and `aria-valuetext` (High/Medium).** No custom slider/grid implements PageUp/PageDown large-step, and `aria-valuetext` is used only by the color widgets. Affected: `slider/slider.tsx:74` (no `aria-valuetext`, name not enforced), `slider/range/use-range-keyboard.ts:24` (no Page keys), ~~`color/color-area.tsx` + `color/color-slider.tsx`~~ **✅** (Page keys + Home/End added), ~~`date-picker/use-date-picker-keyboard.ts:104`~~ **✅** (Page month / Shift+Page year added), `progress/progress-bar.tsx:40` + `progress-gauge.tsx:54` (no `aria-valuetext` for the visible percent; gauge has no indeterminate path).

**I. floating-ui still emits a non-null role on `Popover` (High). ✅ RESOLVED.** `Popover` previously passed `role: 'dialog'` so `useRole` stamped `role="dialog"` + id on the positioning wrapper while `PopoverContent` *also* conditionally rendered `role="dialog"` — labeled → two nested dialogs (outer unnamed); unlabeled → an unnamed wrapper dialog. Now `popover/popover.tsx` passes `role: null` (mirroring `Menu`/`Listbox`/`Combobox`); `PopoverContent`'s inner div is the sole `role="dialog"` and carries a `useId()` `panelId`, and the trigger hand-rolls `aria-controls={open ? panelId : undefined}` pointing at it. A regression test asserts exactly one `role="dialog"`, no role on the wrapper, and trigger↔panel `aria-controls` wiring. *Remaining (separate, advisory):* the trigger still advertises `aria-haspopup="dialog"` even when the content ships unlabeled (a generic surface, no `role="dialog"`); gating that requires the content to register its labelled-ness back to the trigger, or committing to always naming the content — an architectural call left open.
