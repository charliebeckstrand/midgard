# Audit Accessibility

TRIGGER when: the user asks to audit, check, review, or scan for accessibility issues; asks "is this a11y-clean", "any a11y issues", "run a11y on X". Also auto-eligible after `/ui:component:compose` creates a new component.

You are running a static accessibility audit against the project's frontend source files. The audit produces file:line-anchored findings sorted by severity. It does **not** run a headless browser — this is source analysis only.

## Arguments

$ARGUMENTS

Recognized hints:
- A path or component name → audit that scope.
- `--changed` → audit only files in `git diff --name-only` (staged + unstaged).
- No arguments → fall back to `--changed`; if the diff is empty, audit the components directory of every frontend package.

---

## 1. Load the Project Profile

Read `.claude/cache/project-profile.json`. If missing or stale, invoke `/discover --quiet` and re-read.

Filter `packages` to those with `isFrontend: true` and `framework` in (`react`, `next`). If none, stop and tell the user the project has no React/Next frontend packages to audit.

For each qualifying package, pull:
- `framework` — `react` or `next`. Both parse as TS/JSX; `next`-specific checks also apply.
- `componentsDir` — default scope when no argument and no diff.
- `tokensDir` — used for color-contrast checks if it contains color tokens.

---

## 2. Resolve scope

In priority order:

1. **Explicit path or component name** from `$ARGUMENTS` → audit just that.
2. **`--changed`** → take `git diff --name-only HEAD` plus unstaged. Keep only files under a frontend package's `componentsDir` with extension `.tsx` or `.jsx`.
3. **No argument** → first try changed files; if empty, fall back to every file under each frontend package's `componentsDir`.

If a target component name resolves to multiple matches (folders / files), list them and ask the user which to audit.

---

## 3. Parse the targets

Parse `.tsx` / `.jsx` files using a TypeScript/Babel JSX AST. AST parsing is preferred (catches `aria-*` typos and prop-spread issues that regex misses); regex is acceptable as a fallback for quick scans.

For each target file capture: every element node with its tag, attributes (static + dynamic), children outline, and source line. Also capture spread props (`{...rest}`) and flag elements where required ARIA attributes might come in via spread — those become **warnings** rather than blockers since the spread might supply them.

---

## 4. Discover color tokens (optional)

If the package's `tokensDir` is set, scan it for color entries. Recognized shapes:

- A TS/JS file exporting an object whose leaves are CSS color strings (hex / rgb / hsl).
- A CSS file declaring custom properties like `--color-foo: #aabbcc;`.
- A JSON tokens file (W3C design-tokens or Style Dictionary shape) with `"$value"` color leaves.

Build a map `{ tokenName → resolvedColor }`. Resolve aliases when one token references another. Skip color-contrast checks for any token whose value can't be resolved to a concrete color.

If no tokens directory or no color tokens were found, note this and **skip the color-contrast check entirely** rather than guess from hardcoded class strings.

---

## 5. Run the checks

For each parsed element, run every applicable check. Each check produces zero or more findings.

### 5.1. Semantic HTML

- Flag `<div>` or `<span>` with `role` set to `"button"`, `"link"`, `"checkbox"`, `"radio"`, `"switch"`, `"tab"`, `"option"`, `"menuitem"`. The native element exists; use it.
- Flag `<div>` used as a clickable element (has `onClick` / `@click` and no `role`).
- Flag missing landmarks for page-level templates: a page-level component should contain at least one `<main>`. Skip when the file is clearly a sub-component.

### 5.2. ARIA correctness

- Flag invalid `aria-*` names (typos like `aria-lable`, `aria-decribed-by`).
- Flag redundant ARIA on native elements (`<button role="button">`, `<a role="link">`, `<nav role="navigation">`).
- Flag missing required attributes for the declared role: `role="checkbox"` without `aria-checked`, `role="combobox"` without `aria-expanded` + `aria-controls`, `role="tabpanel"` without `aria-labelledby`, `role="dialog"` without `aria-label` or `aria-labelledby`, `role="img"` without `aria-label` or `aria-labelledby`.
- Flag boolean ARIA attributes set to non-boolean string values (`aria-hidden="yes"`).

### 5.3. Keyboard navigation

- Flag elements with click handlers but no keyboard handler when the element is not natively focusable: `<div onClick>` without `tabIndex` and without a key handler.
- Flag positive `tabIndex` (`tabIndex={1}` or higher) — a known anti-pattern.
- Flag custom interactive widgets that lack the expected key handlers for their role (e.g. `role="listbox"` without arrow-key handling, `role="dialog"` without Escape handling). When the framework-native pattern is to delegate this to a hook (e.g. a project `useRoving` equivalent), check that the hook is invoked rather than asserting the handlers inline.

### 5.4. Focus management

- Flag `outline: none` / `outline-none` / `focus:outline-none` without a compensating focus indicator (a `focus:ring-*`, `focus-visible:*` rule, or equivalent).
- Flag dialog/overlay components (`role="dialog"`, project-detected dialog primitives) that don't reference a focus-trap helper from the project's `hooksDir`. If the project has no focus-trap helper, downgrade to a warning rather than blocker.
- Flag overlays missing focus restoration on close (no `onCloseAutoFocus`, no equivalent ref restoration in the cleanup path).

### 5.5. Label association

- Flag `<input>`, `<textarea>`, `<select>` without an accessible name:
  - No `aria-label`,
  - No `aria-labelledby`,
  - No associated `<label htmlFor>` or wrapping `<label>`,
  - No `id` referenced by a sibling label.
- Flag icon-only buttons (`<button>` whose only child is an `<svg>` or icon component) without `aria-label`.

### 5.6. Color contrast (only if tokens discovered)

For each element with both a foreground token and a background token (detected via the project's class/attribute system):

- Resolve both tokens to RGB.
- Compute WCAG contrast ratio.
- Flag pairs below **4.5:1** for normal text, **3:1** for large text (≥18pt or ≥14pt bold), **3:1** for non-text UI components and graphical objects.

If the project's class system makes the token pairing ambiguous (e.g. utility-class soup), report contrast findings as **warnings**, not blockers.

### 5.7. Reduced motion

For animated elements (CSS animations, transition-bearing classes, motion components like `motion.*` from Motion/Framer Motion, React-Transition-Group `<Transition>`):

- Flag any animation that does not have a `prefers-reduced-motion` guard in the same file or in an imported motion utility. The guard can be:
  - A CSS `@media (prefers-reduced-motion: reduce)` block.
  - A framework hook check (e.g. `useReducedMotion()`).
  - A library-level setting in a project motion config.

If the project has a central motion config that already disables animations under `prefers-reduced-motion`, skip this check.

### 5.8. Image / media alternative text

- Flag `<img>` without `alt`. Decorative images must use `alt=""` (allowed) — empty string is fine; missing entirely is a finding.
- Flag `<video>` and `<audio>` without `<track kind="captions">` or an `aria-label`.
- Flag SVG icons used as standalone images without `aria-label` or `role="img"`.

---

## 6. Score severity

Each check declares its severity. Defaults:

| Severity | Default checks |
| --- | --- |
| **Blocker** | missing label on form control; missing `alt` on `<img>`; `role`-required attribute missing; positive `tabIndex`; click handler on non-focusable element with no keyboard handler |
| **Warning** | redundant ARIA; invalid `aria-*` name; focus-indicator removed without replacement; dialog missing focus trap; reduced-motion guard absent |
| **Nit** | semantic-HTML alternatives (`<div role="button">` → `<button>`) when the runtime behavior is otherwise compliant; missing landmarks on a page-level component |

Bump a finding's severity one notch when a project principle explicitly enforces the rule (e.g. `conventions.principles` mentions "every interactive element must be keyboard-operable").

---

## 7. Output

Print one section per audited component, sorted alphabetically:

```
### Widget (packages/design-system/src/components/widget/widget.tsx)

| Severity | Rule | Location | Suggested fix |
| --- | --- | --- | --- |
| blocker | label-association | widget.tsx:42 | wrap the <input> with a <label> or add aria-label |
| warning | focus-indicator | widget.tsx:18 | replace `outline-none` with `focus-visible:ring-2 focus-visible:ring-accent-500` |
| nit | semantic-html | widget.tsx:7 | render a <button> instead of `<div role="button">` |
```

Conclude with a one-line summary:

```
N components audited · M findings (B blockers · W warnings · n nits)
```

If any **blocker** findings exist, mark the audit as **FAIL** at the top of the report. If only warnings/nits exist, mark as **PASS WITH FINDINGS**. If nothing was found, mark as **PASS**.

## Worked examples (fabricated)

### Missing accessible name

Before — icon-only button:

```tsx
<button onClick={onClose}>
  <CloseIcon />
</button>
```

Finding:

```
blocker · label-association · widget.tsx:42 · add aria-label="Close" to the <button>
```

After:

```tsx
<button onClick={onClose} aria-label="Close">
  <CloseIcon />
</button>
```

### Missing focus trap in a dialog

Before — dialog without focus management:

```tsx
export function ConfirmDialog({ open, onClose, children }: Props) {
  if (!open) return null
  return (
    <div role="dialog" aria-labelledby="confirm-title">
      <h2 id="confirm-title">Confirm</h2>
      {children}
      <button onClick={onClose}>Cancel</button>
    </div>
  )
}
```

Finding:

```
warning · focus-trap · confirm-dialog.tsx:5 · wrap the dialog body with the project's focus-trap helper so Tab cycles inside the dialog while open
```

After (assuming the project ships a `useFocusTrap` hook):

```tsx
export function ConfirmDialog({ open, onClose, children }: Props) {
  const ref = useRef<HTMLDivElement>(null)
  useFocusTrap(ref, open)
  if (!open) return null
  return (
    <div ref={ref} role="dialog" aria-labelledby="confirm-title">
      ...
    </div>
  )
}
```

---

## Important

- This skill **reads** source — it never edits. Findings are recommendations; the user decides what to fix.
- Use the project's components and helpers (focus-trap hooks, motion utilities, icon components) when suggesting fixes — pull those names from the profile and from quick scans of `primitivesDir` / `hooksDir`.
- When a check would require browser-only evidence (computed styles, actual focus order, screen-reader output), state the limitation in the rationale rather than guessing.
- If the project has `eslint-plugin-jsx-a11y` configured, note it in the closing summary so the user knows the audit overlap.
- For Next.js apps, also flag missing `next/image` `alt`, missing `next/font` fallback fonts (a CLS / readability concern when fonts fail to load), and `<a>` tags that should be `next/link` (only when navigation matters — pure anchors are fine).
