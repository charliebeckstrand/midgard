# CLAUDE.md

## Principles

- Simplicity above all. The best solution is the simplest one that fully solves the problem. Speculative abstractions age poorly — earn complexity through proven need.
- Hold yourself to a staff engineer standard. Only propose changes you would confidently ship to production. Challenge your own work before presenting it.
- If a solution feels wrong, iterate until it doesn't. Demand excellence.

## Code

- Understand before modifying. Read the surrounding code, follow its conventions, and let consistency guide your decisions.
- Build from small, composable pieces. Colocate what belongs together. Let the type system carry its weight — if a type is hard to express, rethink the design.
- Formatting is tooling's job. Never fight the formatter.
- Solve the stated problem — not adjacent ones. A bug fix is not a refactoring opportunity. If the right fix requires broadening scope, ask first.

## File naming

Filenames must stay legible when stripped of folder context — editor tabs, stack traces, grep results, PR diffs. Every file should be self-identifying anywhere it appears.

Inside `packages/ui/src/components/<name>/` (and the parallel `primitives/`):

- **Main component:** `<name>.tsx` — matches the folder name. Never `component.tsx`.
- **Sub-components:** `<name>-<part>.tsx` — prefixed with the folder name. Never bare (`item.tsx`, `trigger.tsx`). When the folder name is plural, the singular stem is also accepted (`tabs/tab-list.tsx`, `tabs/tab.tsx`) — match the part's component name (`TabList`, `Tab`).
- **Hooks:** `use-<name>-<hook>.ts` (or `.tsx` when the hook returns JSX). The folder name (or its singular stem) appears in every hook filename. Never bare (`hook.ts`, `use-state.ts`).
- **Context:** `context.ts`. Use `.tsx` only when the file exports a provider component containing JSX.
- **Types:** `types.ts` when extracted from the main file.
- **Variants:** `variants.ts` when the recipe / `class-variance-authority` config is extracted alongside the component.
- **Slots:** `slots.tsx` (or `slots.ts`) for components exposing a composable slot API — an intentional, existing pattern.
- **Barrel:** `index.ts`, re-exports only.

When in doubt, prefix with the folder name. Bare filenames read fine inside the folder and turn into noise the moment they appear anywhere else.

Every component or hook file must also export a symbol whose PascalCase (or `useCamelCase`) form matches the filename — `tag-input-badge.tsx` exports `TagInputBadge`, `use-tag-input-keyboard.ts` exports `useTagInputKeyboard`. `packages/ui/scripts/check-component-filenames.ts` enforces this on pre-commit; `packages/ui/scripts/filename-allowlist.json` carries a small set of grandfathered exceptions where renaming would break a stable public API (`Field`, `Label`, `ConfirmDialog`, `QueryRule`, etc.). Never extend that allowlist for new files — fix the file or fix the export.

## Architecture

- Extend before inventing. Prefer growing an existing module over creating a new one unless there is a clear, distinct boundary.
- Dependencies flow inward. Shared packages never depend on application code.
- Abstractions are extracted, not predicted. Duplication across multiple call sites earns a shared utility; a single use case does not.
- When an existing component almost fits but is missing a capability (e.g. ref forwarding, a variant, a prop), ask the user whether extending the component makes sense long-term — not just for the immediate use case. Never update a shared component without explicit approval; propose the change and let the user decide.
- `packages/ui/src/recipes/` is split into a public substrate layer and an internal layer. `ryu/` carries the cross-cutting scales and currents — re-exported from `src/recipes/index.ts` and consumed wherever a substrate is needed. `kata/` (per-component recipes) and `waku/` (control primitives) are internal: they live 1:1 with components, are imported via relative paths from inside the package, and are deliberately not listed in `package.json` `exports`. The boundary is pinned by `src/__tests__/recipes/internal-boundary.test.ts`.

## Git

- Imperative mood, atomic commits. Each commit represents one logical change, described by what it does — not what you did.
- Feature branches for non-trivial work. Never force-push shared branches.
- Review your own diff before committing. Read it as a reviewer would.

## Workflow

For non-trivial work (three or more steps), enter planning mode before writing code. Delegate research to subagents — one focused task per agent — and keep the main context window clean. Summarize at milestones, not line by line.

## Testing

`packages/ui` is the only package with a test runner (`vitest`). The full suite is the lefthook pre-commit gate — that one run is the source of truth for "everything green". **Inside the editing loop, never invoke `pnpm --filter ui test` or `pnpm test`.** Run a scoped subset instead and let lefthook catch anything the scoped run missed at commit time.

- **Known test files** (after editing a component whose test file is obvious): `pnpm --filter ui exec vitest run <test-file>...`. Fastest path — runs only the named files.
- **Unknown tests for a source file**: `pnpm --filter ui test:related <source-file>...`. Walks the dependency graph and runs every test that transitively imports the file.
- **Git diff**: `pnpm --filter ui test:changed`. Runs every test that imports a file in `git status`.

`--changed` and `related` widen to the full suite when the change touches a fan-out file (`recipes/`, `core/`, `primitives/`, a barrel re-exported everywhere). For those, target test files by name. Skills that run tests (`/typescript:review`, `/tests:compose`, `/postmortem`) follow this convention automatically — they only fall back to the full package suite when no scoped command applies.

## Skills

Project-level skills live under `.claude/commands/` and are organized into namespaces (`repo/`, `ui/`, `ui/component/`, `ui/docs/`, `audit/`, `skill/`, `tests/`) plus top-level utilities. Every skill except `/repo:manifest` opens by reading the Manifest tracked at `./manifest.json` (committed to the repo). `/postmortem` and `/premortem` create the file if it is missing; `/postmortem` additionally refreshes it when a diff invalidates it. Consumer skills stop and direct the user to run `/repo:manifest` when the file is absent — they never generate it themselves.

- When asked to discover, profile, fingerprint, or summarize the project, stack, or workspace, always use the `/repo:manifest` skill. It scans the repo (package manager, monorepo tool, frameworks, test runners, conventions docs, components/tokens directories, pre-commit gates, CI provider) and writes a canonical Manifest to `./manifest.json` at the repo root. Other skills consume this manifest instead of re-deriving the same facts.
- When asked to create, add, build, or scaffold a new UI component, always use the `/ui:component:compose` skill. It reads the Manifest, samples sibling components for conventions, runs a composition-first audit, and scaffolds a component matching the discovered styling system and patterns. It audits the target directory to detect the file-split pattern siblings use (component-per-file vs bundled, types inline vs in `types.ts`, hooks inline vs their own files) and matches that pattern, with fallback heuristics when siblings disagree. It does **not** author the matching docs file — it delegates to `/ui:docs:compose` for that, keeping component scaffolding and docs authoring as distinct concerns.
- When asked to recommend, suggest, or identify new UI components — or asked what's missing, what to build next, or how to expand the library — always use the `/ui:component:recommend` skill. After recommendations are approved, use `/ui:component:compose` to create each one.
- When asked to create, add, write, or scaffold a docs page, demo, or example file for a UI component, always use the `/ui:docs:compose` skill. It reads the Manifest, locates the project's docs system (a demos directory, Storybook, an `import.meta.glob`-driven registry, etc.), samples sibling docs files for authoring conventions, and produces a docs page that demonstrates the component's API surface in a form the project's code-derivation tooling can parse. `/ui:component:compose` invokes it automatically when creating a new component in a project with a docs system.
- When asked to audit, check, review, or scan the project's component docs / demos / stories — or asked "are the docs in sync", "any stale demos", "does every component have a docs page" — always use the `/ui:docs:audit` skill. It runs static checks (coverage gaps, registry hygiene, prop-surface sync, code-derivation friendliness, controls reuse, authoring conventions) and produces severity-sorted, file:line-anchored findings.
- When asked to audit, check, review, or scan a UI component — or asked "is this component getting messy", "any bugs in this component", "what should I split", "what should I consolidate" — always use the `/ui:audit` skill. It runs static per-component checks (complexity and size, single-responsibility violations, prop-surface bloat, unused props, dead variants, conditional-rendering smells, state and effect smells, split candidates, consolidate candidates, naming drift, file layout violations against the sibling pattern, framework smells) and produces severity-sorted, file:line-anchored findings. With a component named it audits that one; with no argument it sweeps the suite and ranks worst offenders.
- When asked to create, add, write, or scaffold tests for anything — components, primitives, hooks, utilities, modules — always use the `/tests:compose` skill. It detects the target package via the Manifest, picks patterns matching the discovered test runner and framework, and writes tests that match the codebase's formatting and conventions. `/ui:component:compose` invokes `/tests:compose` automatically when creating a new component.
- When asked to audit, check, review, or scan the project's tests / test coverage / test suite — or asked "are the tests in sync", "any stale tests", "does every component have a test" — always use the `/tests:audit` skill. It runs static checks (coverage gaps, required-pattern drift from `/tests:compose`, runner and helper drift, mock hygiene, source-surface sync, test-layout violations, forbidden patterns, authoring conventions) and produces severity-sorted, file:line-anchored findings.
- When asked to wrap up a task, finish up, or decide what to do before committing — or any time a task is reported complete and a non-empty diff exists — always use the `/postmortem` skill. It reads the diff, classifies the change, and routes to the smallest set of downstream skills the change actually warrants (`/security-review`, `/audit:a11y`, `/tests:compose`, `/simplify`, `/typescript:review`), returning PROCEED for trivial diffs. **Use `/postmortem` as the pre-commit gate — it decides whether `/typescript:review` is necessary; do not invoke `/typescript:review` unconditionally before every commit on your own.**
- When asked to review changes before a commit — "review my diff", "review this TS file", "check this before I commit" — or right after `/ui:component:compose` or `/tests:compose` writes a new `.ts` / `.tsx` file, always use the `/typescript:review` skill. In diff mode it reads the staged diff, runs the project's discovered tests and type-checker for the touched packages, and blocks the commit until findings are resolved; in file mode it vets a freshly-written file against the project's TypeScript principles and advanced-features catalog before the caller declares done. `/postmortem` is the orchestrator that decides when the diff-mode review runs; invoke `/typescript:review` directly only when the user asks for a review explicitly or when a compose skill needs the file-mode pass.
- When asked to recommend, suggest, or identify refactor opportunities — duplication, layering violations, dead code, single-use abstractions, naming inconsistencies, stale markers — always use the `/audit:refactor` skill. It scans the repo with universal heuristics, weighs them against declared principles, and produces a ranked table with file:line evidence.
- When asked to audit, check, review, or scan accessibility — or right after `/ui:component:compose` creates a new frontend component — always use the `/audit:a11y` skill. It runs static a11y checks (semantic HTML, ARIA correctness, keyboard nav, focus management, label association, color contrast against discovered tokens, reduced-motion, alt text) and produces severity-sorted, file:line-anchored findings.
- When asked to audit, scan, or check TypeScript code for smells — `as any`, missed narrowings, hand-rolled exhaustiveness, enum usage, mutable shared data, missed advanced features, convention drift — always use the `/typescript:audit` skill. Pass a file or directory for per-file mode; pass nothing for the project-wide sweep. It produces severity-sorted, file:line-anchored findings against the project's discovered conventions and the principles catalog documented in `/typescript:review`. **Use `/typescript:audit` for periodic clean-up and on-demand polish; use `/typescript:review` for change-driven gating (staged diff, newly-created files).**
- When asked to deliberate, scope, shape, or define a fuzzy idea — or any time the user has a vague starting point they want to turn into something actionable — always use the `/deliberate` skill. It surveys the seed against seven dimensions (outcome, task type, scope, audience, deliverable, constraints, success criteria), asks 3–5 concrete ABCD-style questions only on the dimensions the seed leaves open, then synthesizes the user's picks into a Brief precise enough to hand off without further clarification. **Use `/deliberate` when the question is *what is the work*; use `/council` when the question is *whether* or *which* among known options; use `/premortem` when a plan is already drafted and about to execute.**
- When asked to audit, review, or critique anything — code, design, architecture, plan, decision — always use the `/council` skill. Also use it any time plan mode is entered for non-trivial work (3+ steps, multiple viable approaches, or meaningful tradeoffs); skip it when the path in plan mode is already obvious (single-file edits, mechanical refactors, well-scoped bug fixes). The council runs the question through 5 independent advisors (Contrarian, First Principles Thinker, Expansionist, Outsider, Executor), peer-reviews their responses anonymously, and produces a chairman's verdict with a single concrete next step. Do not use the council for trivial yes/no questions, factual lookups, or casual decisions without meaningful tradeoffs.
- When asked to stress-test, pressure-test, premortem, or check what could go wrong with a drafted plan before executing it — or any time a plan exists in `~/.claude/plans/` and the user is about to start implementation on non-trivial work — always use the `/premortem` skill. It runs the plan through 5 independent failure archetypes (Scope Creeper, Boundary Breaker, Hidden Dependency Hunter, Point of No Return, User-Reality Gap), peer-reviews them anonymously, and produces an Examiner's verdict: most likely failure, falsifiable assumptions to verify, and a concrete diff to the plan. **Use `/council` when the question is *whether* or *which* (decision not yet made); use `/premortem` when the question is *will this plan actually work* (decision made, plan drafted, about to execute).** Skip premortem for trivial single-file edits, mechanical refactors, or when the failure mode is already obvious.
- When asked to compose, scaffold, create, add, draft, or author a new project skill — a file under `.claude/commands/` — or asked how to extend the skill catalog with a new slash command, always use the `/skill:compose` skill. It samples sibling skills to detect the catalog's conventions, scaffolds the new file from the de-facto template, wires the binding into this `## Skills` section, and runs `/skill:audit` on the result before declaring done so the new skill enters the catalog at the same quality bar as the rest.
- When asked to audit, tighten, polish, critique, or improve a project skill — a file under `.claude/commands/` — or asked about trigger overlap, prose quality, Manifest schema gaps, or cross-skill consistency, always use the `/skill:audit` skill. It defaults to auditing the **specific skill** under discussion (recently edited, recently mentioned, or named in the working diff) and falls back to auditing every skill in the catalog when no specific skill is in context. It validates Manifest references against the canonical schema in `/repo:manifest` and produces per-skill findings plus cross-cutting recommendations (schema additions, extraction candidates).
- When asked to recommend, suggest, or identify new project skills to add — what's missing from the catalog, what slash commands to author next, which declared principles still lack an automated skill, or where the namespaces are sparse — always use the `/skill:recommend` skill. It inventories the existing skills under `.claude/commands/`, weighs them against the discovered stack and declared principles, and produces a ranked table of new-skill and extension candidates. After recommendations are approved, hand off to `/skill:compose` for each chosen item.
